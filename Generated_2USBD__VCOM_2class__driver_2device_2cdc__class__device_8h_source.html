71 /*
72  * \note There are several major drawbacks to the CDC-ACM standard USB class,
73  * however it is very standardized and thus usually available as a built-in
74  * driver on most platforms, and so is a better choice than a proprietary
75  * serial class.
76  *
77  * One major issue with CDC-ACM is that it requires two Interface descriptors,
78  * which will upset most hosts when part of a multi-function "Composite" USB
79  * device. This is because each interface will be loaded into a separate driver
80  * instance, causing the two interfaces be become unlinked. To prevent this, you
81  * should use the "Interface Association Descriptor" addendum to the USB 2.0 standard
82  * which is available on most OSes when creating Composite devices.
83  *
84  * Another major oversight is that there is no mechanism for the host to notify
85  * the device that there is a data sink on the host side ready to accept data.
86  * This means that the device may try to send data while the host isn't
87  * listening, causing lengthy blocking timeouts in the transmission routines.
88  * It is thus highly recommended that the virtual serial line DTR (Data
89  * Terminal Ready) signal be used where possible to determine if a host
90  * application is ready for data.
91  *
92  */
